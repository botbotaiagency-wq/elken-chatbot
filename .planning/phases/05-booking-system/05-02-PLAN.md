**Create `lib/booking/slot-checker.ts`:**

Export 3 functions:

1. `checkAndCreateBooking(params)` — calls `supabase.rpc('check_and_create_booking', { p_bot_id, p_facility_type, p_location, p_session_start, p_session_end, p_customer_name, p_member_id, p_contact, p_is_member, p_has_bes, p_gender, p_status })`. Returns `{ success: boolean, booking_id?: string, reason?: string }`.

2. `findNextAvailableSlots(botId, facilityType, location, afterDate, limit=3)` — calls `supabase.rpc('find_next_available_slots', { p_bot_id, p_facility_type, p_location, p_after, p_limit })`. Returns `Array<{ slot_start: string, slot_end: string }>`.

3. `getAvailableSlots(botId, facilityType, location, date)` — queries bookings table to count existing bookings per time slot on a given date, compares against facilities_config capacity, returns list of available `{ slot_start, slot_end }` for that date. Used during the `datetime` step to show the customer available times.

All functions use `createServiceClient()` from `@/lib/supabase/service`.

**Create `lib/booking/state-machine.ts`:**

Export 2 functions:

1. `isBookingExpired(state: BookingState): boolean` — compares `state.last_activity_at` against `Date.now()`. Returns true if gap exceeds 30 minutes (1800000ms).

2. `handleBookingFlow(params: { conversationId: string, botId: string, message: string, state: BookingState | null, detection: { intent: string, language: string }, userId: string, channel: string }): Promise<{ response: string, updatedState: BookingState | null }>` — the main handler.

**handleBookingFlow logic:**

If `state` is null (new booking session):
- Initialize state: `{ step: 'facility', started_at: new Date().toISOString(), last_activity_at: new Date().toISOString() }`
- Return facility selection prompt: a numbered list of facility types using `FACILITY_LABELS`.
  Format: "I'd be happy to help you book a session! Please select a facility type:\n\n1. Bed (Female)\n2. Bed (Male)\n3. Bed (Unisex)\n4. Inhaler\n5. Meeting Room Small (max 8 pax)\n6. Meeting Room Large (max 50 pax)\n\nPlease reply with the number of your choice."

If `state` is non-null:
- Always update `last_activity_at` to now.
- Check if message is off-topic (not a valid selection for the current step). If off-topic AND the message doesn't look like a booking answer (number, date, "yes", "confirm", name, phone number), then:
  - Do NOT clear booking state
  - Call the RAG pipeline (import `detectIntentAndLanguage`, `retrieveContext`, `buildSystemPrompt` from `@/lib/rag/*`) for the off-topic message
  - Get RAG response text (use non-streaming Anthropic call with `claude-haiku-4-5-20251001`, max_tokens 512)
  - Append to response: "\n\nTo continue your booking, please [current step prompt re-stated]"
  - Return the combined response with unchanged booking state

For each step:

**step: 'facility'** — parse user input as number 1-6 or text match against facility labels. If invalid, re-prompt. On valid selection, check if facility has only one valid location (e.g., bed_unisex → subang only, room_small → okr only). If single location, auto-set location, skip to 'datetime' step. Otherwise, move to 'location' step. Update state with `facility_type`.

**step: 'location'** — parse user input as number (1. OKR, 2. Subang) or text match. Only show locations valid for the selected facility type (from `FACILITY_LOCATION_CONSTRAINTS`). On valid selection, move to 'datetime' step.

**step: 'datetime'** — prompt "Please enter your preferred date (e.g., 2026-03-25) or type 'today' or 'tomorrow':" If user provides a date, call `getAvailableSlots(botId, facilityType, location, date)` to get available times. Display available slots as a numbered list with start times. If no slots available, say "No available slots on [date]. Would you like to try another date?" If user selects a slot number, store `session_date` and `session_time`, move to 'details'. Also enforce min_advance_hours and max_window_days cutoffs: fetch from facilities_config, reject dates too soon or too far out.

**step: 'details'** — prompt for all customer details at once:
"Please provide the following details:\n- Full Name:\n- Member ID (if Elken member, or type N/A):\n- Contact Number:\n- Are you an Elken member? (Yes/No):\n- [If bed booking] Do you have a BES device? (Yes/No):"
Parse the response to extract fields. Use a lenient parser: look for patterns like "Name: X", "Member ID: Y", etc. Also accept freeform text where fields are on separate lines.
Apply path branching:
- Meeting Room + Non-member → reject with "Meeting rooms are available to Elken members only. Please provide your Member ID or select a different facility."
- Set `customer_gender` based on facility type (bed_female → 'female', bed_male → 'male', bed_unisex → ask or infer).
Move to 'summary'.

**step: 'summary'** — display full booking summary:
"Here's your booking summary:\n\n- Facility: [label]\n- Location: [label]\n- Date & Time: [date] at [time]\n- Name: [name]\n- Member ID: [member_id or N/A]\n- Contact: [contact]\n- Elken Member: [Yes/No]\n[- BES Device: Yes/No (if applicable)]\n\nPlease type 'confirm' to submit your booking, or 'cancel' to start over."
Wait for explicit "confirm" or "yes" or "cancel".

On **confirm**:
- Call `checkAndCreateBooking(...)` with status 'pending'
- If success: update state to `step: 'confirmed'`. Return confirmation message:
  - Member + Bed/Inhaler: "Your booking has been submitted for staff approval. You will receive a confirmation message once approved."
  - Non-member + Bed/Inhaler: "Thank you! Our specialist will contact you within 24 hours to complete your booking."
  - Meeting Room: "Your booking has been submitted for staff approval. You will receive a confirmation message once approved."
- If slot_full: call `findNextAvailableSlots(...)`, show alternatives, go back to 'datetime' step with message "That slot is no longer available. Here are the next 3 available times:\n[list]\nPlease select a number or enter a different date."
- If gender_conflict: "That time slot already has a booking for a different gender group. Unisex beds cannot mix genders at the same time. Here are alternative slots:\n[list]"
- Clear booking state from conversation.metadata after confirmed step (set metadata.booking to null)

On **cancel**: clear state, return "Booking cancelled. You can start a new booking anytime by asking to book a session."

**State persistence:** After every step result, update `conversations.metadata` via:
```typescript
await supabase
  .from('conversations')
  .update({ metadata: { booking: updatedState }, updated_at: new Date().toISOString() })
  .eq('id', conversationId)
```
Use `createServiceClient()` for this update.

```typescript
// 5b. Check booking state BEFORE intent detection
// If booking is active, route to booking handler directly — even if intent is not book_session
// This prevents intent misclassification of booking responses (e.g., "yes" → general, "3" → unknown)
if (bookingState && !isBookingExpired(bookingState) && (bot as Record<string, unknown>).feature_flags && ((bot as Record<string, unknown>).feature_flags as Record<string, unknown>)?.booking_enabled) {
  const result = await handleBookingFlow({
    conversationId,
    botId,
    message,
    state: bookingState,
    detection: { intent: 'book_session', language: 'en' }, // language doesn't matter for booking prompts
    userId,
    channel,
  })

  // Log the assistant booking response
  const latencyMs = Date.now() - startTime
  await logMessage({
    conversationId,
    botId,
    role: 'assistant',
    content: result.response,
    intent: 'book_session',
    sourceChunks: null,
    ragFound: null,
    latencyMs,
  })

  return new Response(result.response, {
    headers: {
      'Content-Type': 'text/plain; charset=utf-8',
      'X-Conversation-Id': conversationId,
      'X-Intent': 'book_session',
      'X-Language': 'en',
      'X-Rag-Found': 'false',
    },
  })
}

// Also check for expired booking state and clear it
if (bookingState && isBookingExpired(bookingState)) {
  await supabase
    .from('conversations')
    .update({ metadata: {}, updated_at: new Date().toISOString() })
    .eq('id', conversationId)
  // Fall through to normal intent detection — the expired message will be handled below
}
```

**IMPORTANT:** The active booking state check (step 5b) MUST come BEFORE intent detection (step 6). This is because booking answers like "1", "yes", "confirm" would be misclassified by the intent detector. When booking state exists and is not expired, the state machine handles ALL messages. The off-topic sub-handler inside state-machine.ts will call RAG if needed.

The new booking trigger (step 6b) comes AFTER intent detection because it needs to know that the intent is `book_session`.

The chat route returns a plain text Response (not streaming) for booking responses, because booking messages are pre-written prompts, not Claude-generated content. The `X-Intent: book_session` header lets the test console display the correct metadata.
